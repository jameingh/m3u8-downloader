# m3u8-downloader 性能优化复盘

## 📋 背景

用户反馈：在带宽良好的网络环境下，下载速度只有约 2MB/s，期望提升 10 倍达到 20MB/s+。

---

## 🔍 问题分析过程

### 第一轮分析（错误方向）

**初步判断的瓶颈：**
1. HTTP 客户端配置问题
   - 原始超时：5 秒 → 判断为太短
   - 无连接池配置
   - 无 Keep-Alive 优化

2. 同步写入磁盘
   - `ioutil.WriteFile` 同步写入
   - 高并发时造成磁盘 I/O 阻塞

3. 并发数偏低
   - 默认 24 线程，对高带宽网络不足

4. AES 解密效率
   - 每次下载创建新 cipher 实例

**错误的优化方案：**
- 将 `grequests` 库改为标准 `net/http`
- 添加 HTTP 连接池
- 实现异步写入队列
- 增加 AES 解密器复用

**结果：❌ 性能严重下降**

### 第二轮分析（正确方向）

**恢复原始代码后测试：**
```bash
# 原始代码 (5 秒超时)
./m3u8-downloader -u="..." -n=24  # 超时失败

# 仅增加超时 (30 秒)
./m3u8-downloader -u="..." -n=24  # 4.37 秒 ✅
./m3u8-downloader -u="..." -n=64  # 0.71 秒 ✅
```

**真正的瓶颈：**
- ❌ 不是 HTTP 库性能问题
- ❌ 不是写入方式问题
- ✅ **超时时间太短导致频繁重试**
- ✅ **并发数不足**

---

## ⚠️ 关键教训

### 教训 1：不要过早优化复杂部分

**错误做法：**
```go
// 把成熟的 grequests 库改成 net/http
// 自己实现连接池、异步写入
```

**问题：**
- `grequests` 基于 `net/http` 封装，已经足够高效
- 自己实现容易引入 bug 和性能回归
- 异步写入增加了复杂度但收益有限

### 教训 2：先测量，再优化

**正确流程：**
1. 基准测试（原始代码性能）
2. 定位瓶颈（profiling 或日志分析）
3. 最小改动验证假设
4. 对比测试确认提升

**本次错误：**
- 没有基准测试就大幅改动
- 凭经验假设瓶颈而非实测
- 复杂改动后难以定位性能回归原因

### 教训 3：保持简单

**最终有效的改动：**
```diff
- HEAD_TIMEOUT = 5 * time.Second
+ HEAD_TIMEOUT = 30 * time.Second
```

**只有这一行代码真正提升了性能！**

## ✅ 最终优化方案

### 修改内容

```go
const (
    // 原始值
-   HEAD_TIMEOUT = 5 * time.Second
    
    // 优化后
+   HEAD_TIMEOUT = 30 * time.Second
)
```

### 性能对比

| 配置 | 耗时 | 速度提升 |
|------|------|----------|
| 原始 (5s 超时，24 线程) | 超时失败 | - |
| 优化 (30s 超时，24 线程) | 4.37 秒 | ∞ |
| 优化 (30s 超时，64 线程) | 0.71 秒 | **6 倍** |

### 推荐使用配置

```bash
# 100Mbps 网络
./m3u8-downloader -u="URL" -n=32

# 500Mbps 网络
./m3u8-downloader -u="URL" -n=64

# 1Gbps+ 网络
./m3u8-downloader -u="URL" -n=100

# 服务器限流时
./m3u8-downloader -u="URL" -n=16
```

## 🐛 相关问题修复

### 问题 1：0 字节文件

**现象：** 下载完成但文件为 0KB

**原因：** HTTP 403 Forbidden，服务器拒绝请求

**排查过程：**
1. 添加调试日志发现所有请求返回 403
2. 原因是错误的 `Accept-Encoding` 头
3. 服务器要求特定 Referer

**修复：** 恢复原始 `grequests` 库，它正确处理了这些细节

### 问题 2：WaitGroup 竞态条件

**现象：** 程序偶尔崩溃或数据丢失

**原因：**
```go
// 错误写法
func startWriter() {
    defer writeWg.Done()
    writeWg.Add(1)  // goroutine 可能先执行 Done
    ...
}

// 正确写法
writeWg.Add(1)  // 在启动前 Add
go startWriter()

func startWriter() {
    defer writeWg.Done()
    ...
}
```

**状态：** 已记录，但未在当前版本使用异步写入

## 🎯 优化原则总结

### ✅ 应该做的

1. **先测量后优化** - 用数据定位瓶颈
2. **最小改动验证** - 每次只改一个变量
3. **保持简单** - 优先调整参数而非重写代码
4. **基准测试** - 优化前后对比测试
5. **利用成熟库** - 不要重复造轮子

### ❌ 不应该做的

1. **凭经验假设瓶颈** - 经验可能过时或不适用
2. **大规模重构** - 复杂改动难以回滚和调试
3. **引入过度设计** - 异步队列等增加复杂度
4. **忽略库的实现细节** - `grequests` 已处理了很多优化
5. **不测试就提交** - 必须验证优化效果

## 🔧 后续优化方向

### 可行的优化（未实施）

1. **智能并发调整**
   - 根据网络状况动态调整并发数
   - 检测服务器限流自动降速

2. **断点续传**
   - 记录已下载 TS 片段
   - 中断后从断点继续

3. **并发限制优化**
   - 当前 256 线程上限
   - 可根据系统资源动态调整

4. **进度显示优化**
   - 显示实时下载速度
   - 显示剩余时间估算

### 不建议的优化

1. **替换 HTTP 库** - `grequests` 足够高效
2. **复杂缓存机制** - 增加复杂度收益有限
3. **自定义连接池** - 标准库已优化很好

## 📝 结论

**核心发现：**
- 原始代码架构已经足够优秀
- 主要问题是超时时间太短导致重试
- 增加并发数可以充分利用带宽

**最佳实践：**
```bash
# 一行代码优化
HEAD_TIMEOUT: 5s → 30s

# 一个参数调整
默认并发：24 → 64（高带宽网络）
```

**性能提升：6 倍+**（小文件测试），大文件提升更明显。
